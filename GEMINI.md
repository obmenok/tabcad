# Обязательные правила для проекта TabCAD

Этот файл содержит ключевые инструкции по бизнес-логике, UX и архитектуре кода для приложения TabCAD. Любая ИИ-модель, работающая с этим проектом, обязана неукоснительно следовать этим правилам.

## Часть 1: Бизнес-логика и UX (Пользовательские правила)

1. **Точность:** Все параметры геометрии округляются строго до 4 (четырех) знаков после запятой.
2. **Минимальные значения:** Все числовые параметры, если не указано иное, имеют минимальное ограничение `0.01` (запрещены нули и отрицательные значения, ломающие 3D-модель).
3. **Шаг ввода:** Все параметры изменяются стрелочками в инпутах (UI) с шагом `0.01`.
4. **Защита от пустого ввода (Стирание):** При полном удалении значения из инпута пользователем приложение не должно падать. Оно должно держать в памяти последнее введенное (или дефолтное) значение. Если пользователь нажимает Enter или убирает фокус с пустого поля, система должна вернуть последнее корректное значение.
5. **Robustness (Устойчивость):** Общее правило для взаимозависимых инпутов — ИИ должен написать логику, которая подставит за пользователя корректные значения (обрежет по лимитам), чем позволит приложению выпасть в ошибку или сломать рендер.
6. **Круглые таблетки (Ограничение Диаметр vs Толщина):**
   * При уменьшении диаметра (`W`) так, что он становится меньше толщины (`Tt`), толщина автоматически приравнивается к диаметру (`Tt = W`).
   * При этом сначала сжимается Belly Band (`Hb`), пытаясь сохранить выставленные значения глубины (`Dc`) и радиусов чаши.
   * Если Belly Band упирается в минимум (`Hb < 0.01`), то приложение начинает автоматически сжимать глубину чаши (`Dc`) и зависимые от нее радиусы, учитывая все остальные геометрические ограничения.
7. **Связь Глубины и Радиуса:** Глубина чаши (`Dc`) напрямую вычисляет Радиус чаши (`Rc`) и наоборот (двусторонняя связь).
8. **Формула Толщины:** Толщина таблетки (`Tt`) всегда равна: `Belly Band (Hb) + 2 * Глубина чаши (Dc)`. Все эти три параметра жестко зависимы друг от друга.
9. **Влияние Веса:**
   * Вес напрямую влияет на связку: Толщина (`Tt`) и `Belly Band` (`Hb`). 
   * При ручном уменьшении веса сжимается `Belly Band` вплоть до минимально допустимого значения (0.01).
   * Дальнейшее снижение веса за счет изменения геометрии чаши не производится (система останавливается на Hb = 0.01).
   * **Важно:** Изменение веса не затрагивает Глубину чаши (`Dc`) и Радиусы (`Rc`), обеспечивая сохранность формы профиля.
10. **Bevel Angle:** Угол фаски (Bevel Angle) для ВСЕХ форм не может превышать 60 градусов (строгий максимум).

## Часть 2: Архитектура и Код (Технические правила для ИИ)

1. **Единый источник истины (Single Source of Truth):**
   * Запрещено хардкодить "магические числа" (стартовые габариты, плотность) внутри Dash-колбэков.
   * Все дефолтные значения должны импортироваться из словарей в `core/defaults.py`.
2. **Интерфейс (Dash Components):**
   * Запрещено создавать хардкодные `html.Div(dbc.Input(...))` в `components/sidebar.py`. Все поля создаются только через функцию-фабрику `make_input`.
   * Если инпуту нужно новое свойство (например, `disabled`), расширяй аргументы `make_input`.
3. **Dash Callbacks & Race Conditions:**
   * При использовании `allow_duplicate=True` колбэк ОБЯЗАН возвращать `dash.no_update` для тех полей (`Output`), которые математически не изменились. Это предотвращает бесконечные циклы обновлений интерфейса и дергание курсора.
4. **Обработка ошибок:**
   * Категорически запрещено использовать `try...except Exception:` вокруг вызовов `generate_mesh`. Это скрывает ошибки кода (TypeError, NameError, и т.д.).
   * Разрешено отлавливать только ожидаемые проблемы геометрии: `except (ValueError, ZeroDivisionError, OverflowError)`. Баги кода должны приводить к падению в консоль для отладки.
